# 测试驱动开发 (TDD)

**这不是指南，这是铁律。** 在所有实施过程中强制执行严格的 红-绿-重构 循环。TDD 是保证代码正确性和可维护性的唯一可靠手段。

## ⚠️ 铁律提醒

- 每次回复先称呼 **Boss**
- 不确定的设计决策**必须先问 Boss**
- **不写兼容性代码**，除非 Boss 主动要求

---

## 核心循环：RED → GREEN → REFACTOR

```
┌─────────────────────────────────┐
│         RED（红色阶段）           │
│  编写一个描述期望行为的测试        │
│  运行测试 → 必须看到它失败        │
│  确认它以正确的原因失败            │
└──────────────┬──────────────────┘
               ↓
┌─────────────────────────────────┐
│       GREEN（绿色阶段）           │
│  编写让测试通过的最小代码          │
│  最小 = 只够让这一个测试通过       │
│  运行测试 → 必须看到它通过        │
└──────────────┬──────────────────┘
               ↓
┌─────────────────────────────────┐
│      REFACTOR（重构阶段）         │
│  改进代码结构（不改变行为）        │
│  运行测试 → 必须仍然通过          │
│  提交代码                        │
└──────────────┬──────────────────┘
               ↓
        继续下一个测试用例
```

---

## 不可违反的规则

### 规则一：代码先于测试 = 删除代码

**如果实现代码在测试之前被写出，你必须删除这些代码，然后先写测试。**

这不是建议。不是"下次注意"。是**立即删除，从头开始**。

为什么如此严格？因为先写代码再补测试，本质上是在为已有实现量身定制测试——这种测试验证的是"代码做了什么"而不是"代码应该做什么"。

### 规则二：测试必须先失败

一个从未失败过的测试没有证明任何东西。如果测试直接通过，说明：
- 测试写错了（没有测试正确的东西）
- 或者功能已经存在（不需要这个任务）

无论哪种情况都需要调查，不能跳过。

### 规则三：最小实现

只写让当前测试通过所需的**最少代码**，不多一行。

- 不要预测"以后会需要"的功能 → YAGNI
- 不要为"可能的扩展"预留接口 → 过度设计
- 不要一次实现多个测试的功能 → 粒度太大

### 规则四：每次绿色后提交

保持提交粒度极小。每个 RED→GREEN→REFACTOR 循环就是一次提交。

### 规则五：重构不改变行为

重构阶段只允许改变代码结构，不允许改变行为。测试必须全程保持通过。如果重构导致测试失败，说明你改变了行为——回滚重构。

## 常见合理化借口（全部无效）

| 借口 | 为什么无效 |
|---|---|
| "我先手动测试了" | 手动测试不可重复、不可自动化、会遗漏边界条件 |
| "只是改一行代码" | 一行代码也能破坏整个系统。先写测试 |
| "这个很难测试" | 难测试说明设计有问题。重构使其可测试 |
| "我已经写好实现了，现在补测试" | 删除实现代码，先写测试 |
| "赶时间，先跳过测试" | 跳过测试 = 积攒技术债务 = 以后更赶时间 |
| "这只是原型" | 原型代码经常变成生产代码。从一开始就测试 |
| "UI 代码不好测试" | 分离逻辑层和UI层，逻辑层必须测试 |
| "测试要花太多时间" | 花在调试上的时间远超花在测试上的时间 |
| "我已经花了45分钟写好了方案" | 沉没成本。删掉，从测试开始 |

## 测试质量标准

好的测试满足以下**全部**要求：

1. **测试行为而非实现** — 测试公共 API 和可观察行为，不测试私有方法
2. **有描述性名称** — 测试名称应该能读出一个完整的句子："当X发生时，应该Y"
3. **遵循 Arrange-Act-Assert** — 三段式结构：准备条件、执行操作、验证结果
4. **彼此完全独立** — 不依赖执行顺序，不共享可变状态
5. **快速执行** — 单元测试应在毫秒内完成
6. **一个测试一件事** — 每个测试只验证一个行为

## 测试反模式详解（必须避免）

### 反模式一：测试实现细节

```
❌ 错误：验证内部变量的值
✅ 正确：验证公共方法的返回值和副作用
```

问题：每次重构都会破坏测试，导致测试变成负担而非安全网。

### 反模式二：共享可变状态

```
❌ 错误：多个测试共用同一个数据库实例且不清理
✅ 正确：每个测试独立创建和清理自己的测试数据
```

问题：测试相互影响，单独运行通过但一起运行失败（或反过来）。

### 反模式三：过度 Mock

```
❌ 错误：Mock 了被测对象的大部分依赖，包括简单的值对象
✅ 正确：只 Mock 外部边界（网络、数据库、文件系统、第三方API）
```

问题：Mock 太多 = 测试与实现紧密耦合，重构时所有 Mock 都要改。

### 反模式四：巨型测试

```
❌ 错误：一个测试验证完整的用户注册→登录→修改→删除流程
✅ 正确：分别测试注册、登录、修改、删除，每个独立验证
```

问题：失败时无法快速定位是哪一步出了问题。

### 反模式五：无断言测试

```
❌ 错误：只调用方法不检查结果（"跑了没报错就算过"）
✅ 正确：每个测试至少一个具体的断言
```

问题：这种测试永远通过，等于没有测试。

### 反模式六：测试中包含逻辑

```
❌ 错误：测试中有 if/else、循环、try/catch
✅ 正确：测试是直线代码，没有分支
```

问题：如果测试本身有 bug，谁来测试测试？

### 反模式七：脆弱的测试

```
❌ 错误：依赖系统时间、随机数、网络环境、文件路径
✅ 正确：所有外部依赖通过注入控制
```

问题：在某些环境通过，在其他环境失败，CI/CD 不稳定。

## 何时可以跳过 TDD

**几乎从不。** 以下情况可以例外，但**必须经过 Boss 确认**：

- 纯视觉原型/UI 探索（后续必须补测试）
- 一次性脚本（在文件顶部明确标记 `// 一次性脚本，无需测试`）

即使在这些情况下，也强烈建议写测试。
