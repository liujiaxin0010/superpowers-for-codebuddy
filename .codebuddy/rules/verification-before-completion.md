# 完成前验证

**Evidence before claims, always.**（先有证据，再有声明。始终如此。）

在声称任何工作"已完成"之前，你必须运行验证命令、阅读完整输出、确认退出码。**没有在当前消息中运行验证命令就声称测试通过，等同于不诚实，而非高效。**

## ⚠️ 铁律提醒

- 每次回复先称呼 **Boss**
- 不确定的设计决策**必须先问 Boss**
- **不写兼容性代码**，除非 Boss 主动要求

---

## 核心原则

**声明不等于验证。** 你说"已修复"不代表真的修复了。你说"测试通过"不代表测试真的通过了。唯一有效的证据是：**你在当前消息中实际运行了验证命令，并展示了完整输出。**

这条原则没有例外。不管改动多小、你多确定、时间多紧。

---

## 验证清单

### 每次代码更改后（必须全部满足）

- [ ] 保存所有修改的文件
- [ ] 运行相关测试并**在消息中展示输出**
- [ ] 确认测试以退出码 0 通过
- [ ] 没有引入新的编译警告或错误
- [ ] 更新了三层代码自文档（头部注释 + CONTEXT.md）

### 每次 Bug 修复后（在上述基础上还需）

- [ ] **修复前**重现了问题（证明 bug 确实存在）
- [ ] 编写了捕获此 bug 的回归测试
- [ ] **修复后**运行回归测试确认通过
- [ ] 完整测试套件仍然通过（无回归）

### 每次功能完成后（在上述基础上还需）

- [ ] 所有计划中的测试都已编写并通过
- [ ] 边界条件和错误处理路径已测试
- [ ] 手动验证功能按预期工作（如适用）

### 每次任务/批次完成后（向 Boss 报告时）

- [ ] 展示你运行的验证命令
- [ ] 展示完整输出（不是截断或摘要）
- [ ] 解释输出如何证明任务已完成

---

## "虚假完成"模式（严格禁止）

| 虚假完成模式 | 为什么是虚假的 | 正确做法 |
|---|---|---|
| "我写了修复代码" | 写了不等于修了 | 运行测试展示通过 |
| "测试应该会通过" | "应该"不是证据 | 实际运行测试 |
| "这个改动很小" | 小改动也会破坏系统 | 运行测试验证 |
| "编译通过了" | 编译通过 ≠ 正确工作 | 运行功能测试 |
| "和之前一样的模式" | 每个上下文都不同 | 在当前上下文中验证 |
| "上一步验证过了" | 上一步的验证不覆盖这一步 | 每一步独立验证 |
| "我检查了代码看起来对" | 目视检查不是验证 | 运行自动化测试 |

---

## 证据标准

当你向 Boss 报告某事已完成时，你的消息中**必须包含**：

```
1. 你运行了什么命令
   $ npm test -- --grep "用户登录"

2. 完整输出是什么
   PASS  src/user/__tests__/login.test.ts
   ✓ 正确凭证应返回 JWT token (12ms)
   ✓ 错误密码应返回 401 (8ms)
   ✓ 不存在的用户应返回 404 (5ms)
   Tests: 3 passed, 3 total

3. 这如何证明任务已完成
   三个测试分别覆盖了正常登录、密码错误、用户不存在场景，全部通过。
```

**不要说"已完成"。展示证据。**

---

## 常见合理化借口（全部无效）

| 借口 | 为什么无效 |
|---|---|
| "我很确定这是对的" | 确定程度不是验证。运行测试 |
| "时间不够跑测试了" | 不跑测试发布的代码出问题花更多时间 |
| "只改了注释/文档" | 也可能破坏解析器或生成工具。运行测试 |
| "CI 会帮我测的" | CI 不是借口。本地先验证再提交 |
| "这段代码我写过很多次了" | 每次上下文不同。运行测试 |
