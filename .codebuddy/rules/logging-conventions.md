# 日志规范（强制）

## 目标

保证日志在可观测性、可检索性、可运营性上的一致性，避免“每个改动各写各的日志”。

## 适用范围

适用于业务代码与服务运行时日志（后端、前端、任务进程、脚本服务）。

不适用于以下场景：

1. 命令行工具本身的标准输出结果（输出即产品能力）
2. 用户明确要求使用控制台输出

## 强制规则

1. **沿用项目现有日志结构**
   - 修改已有项目前，必须先阅读现有日志实现：框架、字段、级别、上下文注入方式。
   - 新增日志必须与项目现有结构一致（例如字段名、traceId 传递、模块标识、错误码模式）。

2. **禁止中文日志内容**
   - 日志 message、日志 key、错误描述统一使用英文。
   - 用户可见文案（UI 提示、接口返回文案）可按产品要求，不等同于日志。

3. **默认禁止控制台日志**
   - 禁止在提交代码中保留 `console.log`、`print`、`fmt.Print*`、`System.out*`、`e.printStackTrace()` 等控制台输出。
   - 必须使用项目日志框架/封装（logger、log facade、structured logger）。
   - 仅当用户明确要求或场景确实是 CLI 标准输出时可例外。

4. **新增项目必须先选日志方案**
   - 在 `/brainstorm` 阶段明确日志框架与字段规范。
   - 至少确认：日志级别策略、结构化字段、trace/request id、错误码、脱敏策略、日志滚动策略。

5. **禁止敏感信息入日志**
   - 禁止记录密码、token、密钥、证件号、银行卡号、隐私原文等敏感数据。
   - 必要时仅记录脱敏或摘要信息。

## 执行要求

1. `/spec-lite` 必须在“需求澄清结论”中明确日志策略（沿用或选型）。
2. `/write-plan` 必须包含日志改造与日志验证任务。
3. `/code-review` 必须检查：
   - 是否复用项目日志结构
   - 是否存在中文日志
   - 是否残留控制台输出

## 快速自检建议

在提交前可用以下检索检查是否残留控制台输出（按技术栈调整）：

```bash
rg -n "console\\.log|print\\(|fmt\\.Print|System\\.out|printStackTrace" src
```
