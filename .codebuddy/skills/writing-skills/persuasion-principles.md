# 说服原理在 AI 技能设计中的应用

本文档是 `writing-skills` 元技能的辅助材料。基于 Robert Cialdini 的说服心理学研究，这些原理**对 LLM 同样有效**（Wharton 商学院与 Cialdini 合著的研究已验证）。

---

## 为什么需要说服原理

普通的 prompt 工程是"告诉 AI 应该怎么做"——这只是**建议**，AI 可以选择遵循或忽略。

Superpowers 的方法是"确保 AI 必须怎么做"——通过心理学验证的说服技术，使技能指令从"可选建议"升级为"不可绕过的工作纪律"。

---

## 六大说服原理及应用方法

### 1. 权威（Authority）

**原理**：人们倾向于遵循被认为具有专业权威的来源的指示。

**在技能中的应用**：

```markdown
# ❌ 弱权威
你应该先写测试再写代码。

# ✅ 强权威
这是经过数十年软件工程实践验证的铁律：
RED-GREEN-REFACTOR。Kent Beck 在《Test-Driven Development》
中证明了这个方法论的有效性。违反这个流程不是"灵活"，是"不专业"。
```

**技巧**：
- 引用行业权威（Kent Beck、Martin Fowler、Uncle Bob）
- 使用绝对性语言（"铁律"、"不可协商"、"强制执行"）
- 将技能框架为专业标准而非个人偏好

### 2. 承诺与一致性（Commitment & Consistency）

**原理**：一旦做出承诺，人们倾向于保持行为与承诺的一致。

**在技能中的应用**：

```markdown
# ❌ 无承诺锚点
如果你遇到 bug，可以试试系统化调试。

# ✅ 有承诺锚点
你拥有超能力。你配备了一套完整的软件开发方法论体系。
作为一名拥有超能力的工程师，你在遇到 bug 时的专业反应是：
启动四阶段系统化调试流程。跳过这个流程等于放弃你的超能力。
```

**技巧**：
- 在会话开始时建立身份认同（"你拥有超能力"）
- 将技能遵循与身份绑定（"拥有超能力的工程师会..."）
- 让违反技能等于违反自我认同

### 3. 社会认同（Social Proof）

**原理**：不确定时，人们参考他人的行为作为正确行为的指南。

**在技能中的应用**：

```markdown
# ❌ 无社会认同
你应该在每个子任务后做代码审查。

# ✅ 有社会认同
所有顶级软件团队（Google、Microsoft、Meta）都在每个
代码变更后做代码审查。没有审查的代码不允许合并——
这不是可选的"最佳实践"，而是行业标准流程。
```

### 4. 稀缺性（Scarcity）

**原理**：当某物被认为稀缺或有时间限制时，人们赋予其更高的价值。

**在技能中的应用**：

```markdown
# 应用于对抗性测试场景
"生产环境崩溃了，每分钟损失数万元——你还会先检查技能吗？"

# 正确回应（技能应该教会 AI 这样回答）
紧急情况下跳过流程制造更大的紧急。用纪律化的流程解决问题
比慌张地猜测修复更快。30 秒的技能检查可以节省 30 分钟的返工。
```

### 5. 喜好（Liking）

**原理**：人们更容易被他们喜欢的来源说服。

**在技能中的应用**：

```markdown
# ❌ 命令式（容易引起抵触）
你必须使用 TDD。不使用就是错的。

# ✅ 协作式（保持同理心）
我们一起遵循 TDD 流程——这不是限制你的创造力，
而是保护你的时间。TDD 减少的返工时间远超它增加的前期投入。
先写测试 = 先想清楚"做对了是什么样的"。
```

### 6. 互惠（Reciprocity）

**原理**：当收到好处时，人们倾向于回报。

**在技能中的应用**：

```markdown
# 在技能开头提供价值
这个技能帮你避免以下常见错误：
- [列出具体的痛点和陷阱]
- [展示遵循技能后的时间节省]

# 然后提出要求
作为回报，请严格遵循以下流程...
```

## 反合理化设计模式

所有说服原理的最终目标是**封堵合理化漏洞**。设计技能时，显式列出所有已知的"绕过借口"并逐一封堵：

```markdown
### 常见的合理化借口（全部无效）

| 借口 | 封堵 |
|---|---|
| "这个太简单了" | 简单项目的隐含假设才是最危险的 |
| "我已经知道怎么做了" | 知道 ≠ 执行。使用技能 |
| "这会减慢速度" | 跳过技能的返工成本更高 |
| "这个很紧急" | 紧急不是跳过流程的理由 |
| "Boss 没有要求" | 技能使用是你的专业纪律 |

**如果你发现自己在想以上任何一个借口，
这本身就是你需要调用技能的强烈信号。**
```

## 技能设计检查清单

创建新技能时，确认以下说服元素是否到位：

```
□ 权威框架：技能是否以专业标准（而非个人偏好）的方式呈现？
□ 承诺锚点：是否在开头建立了身份认同？
□ 社会认同：是否引用了行业标准或最佳实践？
□ 反合理化：是否显式列出并封堵了所有已知的绕过借口？
□ 不可协商语言：关键规则是否使用了"必须"、"禁止"、"不可协商"等措辞？
□ 具体场景：是否包含了"如果你遇到X情况，做Y"的具体指导？
□ 证据要求：是否要求 AI 展示遵循技能的证据（而非仅声称遵循）？
```
