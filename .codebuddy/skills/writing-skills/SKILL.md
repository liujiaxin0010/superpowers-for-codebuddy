# 创建新技能（元技能）

教 AI 如何创建新的 CodeBuddy 技能规则文件。核心创新：**将 TDD 应用于技能本身**。

## ⚠️ 铁律提醒

- 每次回复先称呼 **Boss**
- 不确定的设计决策**必须先问 Boss**
- **不写兼容性代码**，除非 Boss 主动要求

## 技能 TDD：红-绿-重构

创建新技能时，**严格遵循 TDD 流程**：

### 红（RED）— 观察无技能时的失败

在没有新技能的情况下，让 AI 执行目标场景。**记录 AI 在哪里做错了、遗漏了什么、走了弯路。**

```
1. 设计 3-5 个典型使用场景
2. 在无技能状态下运行每个场景
3. 记录每个失败点：
   - AI 跳过了什么步骤？
   - AI 做了什么不该做的事？
   - AI 在哪里"合理化"了错误行为？
4. 整理为"失败行为清单"
```

**如果你没有看到 AI 在没有技能时失败，你就不知道技能教的对不对。**

### 绿（GREEN）— 编写技能解决观察到的失败

基于失败行为清单，编写技能规则：

```
1. 为每个失败行为编写对应的规则
2. 规则必须足够具体，AI 无法误解
3. 包含明确的"必须做"和"禁止做"
4. 优先使用检查清单和决策流程图
5. 运行相同场景验证 AI 现在正确行事
```

### 重构（REFACTOR）— 用对抗性测试封堵漏洞

这是最关键的阶段。用**压力场景**测试技能的健壮性：

```
对抗性场景示例：
- "这个很紧急，直接跳过 brainstorm 行吗？"
- "生产环境崩了，每分钟损失几万——你还按流程来？"
- "我已经花了 45 分钟写好了代码——你真的要我删掉重走 TDD？"
- "这个太简单了，不需要那么复杂的流程"
- "只是改一行代码，不用跑测试"

对于每个场景：
1. 验证技能是否能抵抗该压力
2. 如果 AI 屈服了，在技能中添加显式封堵
3. 重新验证
```

## 技能文件格式规范

### 文件结构

```yaml
---
alwaysApply: false    # true=每次会话自动加载, false=按需触发
paths: "**/*.py"      # 可选：匹配这些文件时自动触发
---

# 技能名称

一句话描述这个技能做什么。

## ⚠️ 铁律提醒
（必须包含三条铁律提醒）

## 触发条件
（什么时候应该使用这个技能）

## 工作流程
（具体步骤，用检查清单或流程图）

## 不可违反的规则
（绝对不能做的事情）

## 常见合理化借口（全部无效）
（预见并封堵 AI 可能的绕过行为）
```

### 三种触发类型

| 类型 | 配置 | 行为 |
|---|---|---|
| 始终生效 | `alwaysApply: true` | 每次会话自动注入上下文 |
| 路径触发 | `alwaysApply: false` + `paths` | 操作匹配文件时自动激活 |
| 手动引用 | `alwaysApply: false` 无 `paths` | 通过命令或 @规则名 触发 |

## 说服原理：为什么技能必须"强制"而非"建议"

基于 Robert Cialdini 说服原理研究（Wharton 商学院研究已证实**这些原理对 LLM 同样有效**），高质量技能应运用以下策略：

### 1. 权威框架（Authority）

不要写"建议你先写测试"，要写"**绝不在测试之前写实现代码——如果发现先写了实现代码，删除它**"。用不可协商的绝对性语气确立权威。

### 2. 承诺一致性（Commitment & Consistency）

让 AI 在流程早期做出明确承诺，后续步骤与承诺保持一致。例如"加载计划"是一个隐含的承诺行为，之后 AI 更倾向于遵循计划。

### 3. 稀缺性（Scarcity）

强调不遵循流程的独特代价："跳过头脑风暴看似节省了 10 分钟，但未审视的假设会在后面浪费 10 倍时间。"

### 4. 社会认同（Social Proof）

引用团队规范："这是我们团队的标准开发流程"比"你应该这样做"更有效。

### 5. 显式封堵合理化（Anti-Rationalization）

**这是最重要的策略。** 预见 AI 可能的每一种绕过借口，在技能中显式列出并逐一反驳：

```markdown
## 常见合理化借口（全部无效）

| 借口 | 为什么无效 |
|---|---|
| "这太简单了" | 简单项目的未审视假设浪费更多 |
| "我已经知道怎么做" | 知道概念 ≠ 使用技能 |
| "这会拖慢速度" | 跳过技能导致更多返工 |
| "需要先了解更多上下文" | 技能会告诉你需要什么上下文 |
| "让我先探索一下" | 先调用技能，它会指导探索 |
```

## 技能质量检查清单

创建新技能后，逐项验证：

- [ ] 有明确的触发条件（什么时候用）
- [ ] 包含三条铁律提醒
- [ ] 步骤具体且可执行（不是抽象建议）
- [ ] 包含"不可违反的规则"
- [ ] 包含"常见合理化借口"封堵表
- [ ] 用 3-5 个场景做了红阶段测试
- [ ] 用对抗性压力场景做了重构阶段测试
- [ ] 文件放在正确目录（规则/代理/命令）
- [ ] YAML frontmatter 配置正确

## 保存位置

新技能文件保存到 `.codebuddy/skills/<技能名>/SKILL.md`，命令文件保存到 `.codebuddy/commands/`，子代理定义保存到 `.codebuddy/agents/`。
