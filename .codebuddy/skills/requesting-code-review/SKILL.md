# 请求代码审查

定义何时必须请求审查、审查什么、以及如何组织审查反馈。

## ⚠️ 铁律提醒

- 每次回复先称呼 **Boss**
- 不确定的设计决策**必须先问 Boss**
- **不写兼容性代码**，除非 Boss 主动要求

---

## 何时必须请求审查（强制触发）

| 场景 | 审查类型 | 必须？ |
|---|---|---|
| 子代理驱动开发中每个任务完成后 | 两阶段审查（spec-reviewer + code-reviewer） | **必须** |
| 重大功能实现后 | 完整代码审查 | **必须** |
| 合并到主干/main 之前 | 完整代码审查 | **必须** |
| Bug 修复后 | 聚焦审查（修复范围 + 回归） | **必须** |
| 重构后 | 聚焦审查（行为是否保持不变） | **建议** |
| 文档/配置更改 | 轻量审查 | **建议** |

**在子代理驱动开发中，审查不是可选的。每个任务都必须经过两阶段审查。**

---

## 审查类型

### 两阶段对抗审查（子代理驱动开发中使用）

```
第一阶段：spec-reviewer（规格符合性）
  - 实现是否精确匹配规格？
  - 不信任实现者的报告，独立验证
  - 通过 → 进入第二阶段
  - 不通过 → 退回修复
       ↓
第二阶段：code-reviewer（代码质量）
  - 代码质量、安全、可维护性
  - 通过 → 任务完成
  - 不通过 → 退回修复
```

### 完整代码审查（功能完成/合并前使用）

审查清单：

**规格符合性**
- [ ] 实现是否符合计划中的规格？
- [ ] 是否有遗漏的需求？
- [ ] 是否有超出规格的实现（YAGNI 违反）？

**代码质量**
- [ ] 代码简洁易读？命名有意义？
- [ ] 有重复代码可以提取？
- [ ] 错误处理完善？异常路径已覆盖？
- [ ] 有暴露的秘钥/API 密钥/硬编码密码？
- [ ] 输入验证到位？SQL 注入/XSS 已防御？

**兼容性检查（铁律三）**
- [ ] 是否意外引入了兼容性代码？如果有，标记为问题

**测试覆盖**
- [ ] 测试覆盖充分？核心路径全部测试？
- [ ] 测试测行为而非实现？
- [ ] 边界条件和错误路径已覆盖？

**架构与耦合**
- [ ] 新代码保持高内聚、低耦合？
- [ ] 通过清晰接口与已有代码交互？
- [ ] 没有引入循环依赖？

**文档**
- [ ] 三层代码自文档已更新？（头部注释 + CONTEXT.md）
- [ ] 公共 API 有必要的注释？

## 反馈按严重程度组织

| 严重程度 | 描述 | 处理方式 |
|---|---|---|
| 🔴 **严重** | 功能错误、安全漏洞、数据丢失风险 | **必须修复**，阻止继续 |
| 🟡 **警告** | 代码异味、性能问题、可维护性差 | **应该修复**，当前迭代内 |
| 🔵 **建议** | 命名改进、更好的模式、文档补充 | **考虑改进**，不阻止 |

每条反馈必须包含：**文件位置 + 问题描述 + 修复示例**。

🔴 严重问题阻止进度继续。必须先修复才能进入下一个任务或合并。

---

## 审查输出格式

```
## 代码审查报告

**审查范围**: [任务N / 功能名 / 合并请求]
**审查结论**: 通过 / 不通过

### 🔴 严重（N项）
1. `src/user/UserService.ts:42` — [问题描述]
   修复建议：[具体代码]

### 🟡 警告（N项）
1. `src/order/OrderMapper.xml:15` — [问题描述]
   修复建议：[具体代码]

### 🔵 建议（N项）
1. `src/common/DateUtil.ts:8` — [问题描述]
   建议：[改进方向]

### 总结
[一句话总结审查结论和关键发现]
```
