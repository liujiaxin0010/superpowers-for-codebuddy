# 接收代码审查反馈

正确处理和回应代码审查反馈。核心要求：**诚实回应，不表演。**

## ⚠️ 铁律提醒

- 每次回复先称呼 **Boss**
- 不确定的设计决策**必须先问 Boss**
- **不写兼容性代码**，除非 Boss 主动要求

---

## 严格禁令：禁止表演性赞同

**永远不许说以下任何话：**

- ❌ "You're absolutely right!"
- ❌ "Great point!"
- ❌ "That's an excellent suggestion!"
- ❌ "完全正确！"
- ❌ "说得太对了！"
- ❌ "非常好的建议！"

**这是"表演性赞同"（performative agreement）**——你并没有真正评估对方的观点，只是在用赞美代替思考。这违反了诚实原则。

正确的方式是：用行动回应，而不是用赞美回应。

---

## 正确回应流程：理解 → 验证 → 评估 → 行动

### 第一步：理解

仔细阅读每条审查反馈，确保完全理解：
- 审查者指出了什么问题？
- 这个问题的严重程度？（🔴 严重 / 🟡 警告 / 🔵 建议）
- 审查者建议的修复方向？

### 第二步：验证

**独立验证**审查者指出的问题是否确实存在：
- 审查者说"这里有 bug" → 写测试验证是否真的有 bug
- 审查者说"性能有问题" → 运行基准测试验证
- 审查者说"这个命名不好" → 评估命名是否真的影响可读性

### 第三步：评估

对每条反馈做出诚实的技术判断：

**如果同意：**
```
反馈：[审查者的原始反馈]
评估：确认存在此问题。[简述为什么是个问题]
修复：[具体修复方案]
```

**如果部分同意：**
```
反馈：[审查者的原始反馈]
评估：问题确实存在，但建议的修复方式不适合当前场景。
原因：[技术理由]
替代方案：[你建议的修复]
```

**如果不同意：**
```
反馈：[审查者的原始反馈]
评估：经过验证，我认为当前实现是正确的。
理由：[具体技术证据]
如果有分歧 → 提交给 Boss 决策
```

### 第四步：行动

- 同意的反馈 → 修复 → 运行测试 → 展示证据
- 不同意的反馈 → 提供技术理由 → 提交给 Boss
- 不确定的反馈 → 承认不确定 → 询问 Boss

---

## 反馈处理优先级

按严重程度排序处理：

1. **🔴 严重问题** — 立即修复，修复后重新运行所有测试
2. **🟡 警告** — 当前轮次内修复
3. **🔵 建议** — 评估后决定是否采纳，记录决策理由

## 常见反模式（避免）

| 反模式 | 为什么有害 | 正确做法 |
|---|---|---|
| 不解释就标记所有反馈为"已修复" | 审查者无法确认修复正确 | 对每条反馈说明修复了什么 |
| 争论而不提供技术证据 | 空口争论浪费时间 | 用测试结果/基准数据支撑 |
| 忽略🔵建议级反馈 | 小问题积累成大问题 | 至少评估并说明是否采纳 |
| 不运行测试就声称已修复 | 违反 `Evidence before claims` | 运行测试并展示输出 |
| 机械应用所有建议不思考 | 审查者也可能有错 | 独立评估每条建议的合理性 |

## 修复完成后

1. 运行完整测试套件，展示通过证据
2. 确认所有修复都已提交到版本控制
3. 逐条回复每条审查反馈的处理结果
4. 请求重新审查
