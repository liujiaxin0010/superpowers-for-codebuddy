# 项目功能扩展

在已有项目上安全地新增功能，做到**不影响原有功能**、**高内聚、低耦合**。

## 触发条件

当 Boss 要求对已有项目增加新功能时激活，或通过 `/extend` 命令触发。

## ⚠️ 铁律提醒

- 每次回复先称呼 **Boss**
- 不确定的设计决策**必须先问 Boss**
- **不写兼容性代码**，除非 Boss 主动要求

---

## 第一阶段：项目理解（必须完成，不可跳过）

### 1.1 全局结构扫描

```bash
# 扫描项目目录结构（排除依赖目录）
find . -type f -name "*.java" -o -name "*.py" -o -name "*.ts" -o -name "*.js" -o -name "*.go" -o -name "*.vue" -o -name "*.jsx" -o -name "*.tsx" -o -name "*.cpp" -o -name "*.c" -o -name "*.rs" -o -name "*.rb" | grep -v node_modules | grep -v vendor | grep -v __pycache__ | grep -v .git | grep -v .svn | head -200

# 查看项目配置文件
cat package.json 2>/dev/null || cat pom.xml 2>/dev/null || cat build.gradle 2>/dev/null || cat requirements.txt 2>/dev/null || cat go.mod 2>/dev/null || cat Cargo.toml 2>/dev/null || echo "未检测到标准项目配置"
```

### 1.2 架构模式识别
必须识别并向 Boss 汇报：
- **项目类型**：Web 前端 / 后端 API / 桌面应用 / CLI 工具 / 库 / 全栈 等
- **技术栈**：语言、框架、主要依赖
- **架构模式**：MVC / MVVM / 分层架构 / 微服务 / 单体 / 插件式 等
- **目录约定**：代码组织方式（按功能、按层、按模块）
- **现有测试**：测试框架、测试覆盖情况、测试运行命令
- **构建方式**：构建工具、构建命令、部署方式

### 1.3 关键入口和依赖图
- 找到主入口文件
- 梳理核心模块之间的依赖关系
- 识别公共工具/基础设施模块
- 识别配置文件和环境变量

### 1.4 编码风格识别
通过阅读现有代码，识别并遵循：
- 命名规范（camelCase / snake_case / PascalCase）
- 缩进风格（空格数 / Tab）
- 注释风格
- 文件组织惯例
- 错误处理模式
- 日志记录方式

日志记录方式必须进一步识别：

1. 日志框架/封装入口（logger 包、门面、拦截器）
2. 日志字段结构（traceId、模块、动作、结果、错误码、耗时等）
3. 日志级别使用约定
4. 是否存在中文日志（新改动不得引入中文日志）
5. 控制台输出使用边界（默认禁止）

### 1.5 输出：项目理解报告
完成以上分析后，向 Boss 提交一份简洁的项目理解报告：

```markdown
## 项目理解报告

**项目类型**: [类型]
**技术栈**: [语言 + 框架 + 关键依赖]
**架构模式**: [模式]
**目录结构约定**: [简要说明]
**测试现状**: [框架 + 覆盖情况 + 运行命令]
**编码风格**: [关键特征]
**核心模块**: [列出核心模块及其职责]
```

等待 Boss 确认理解是否正确后，再进入下一阶段。

## 第二阶段：影响评估

### 2.1 确认新功能需求
向 Boss 逐一确认：
- 新功能的具体需求是什么？
- 预期的用户交互方式是什么？
- 有没有参考示例？
- 优先级和范围边界在哪？

### 2.2 影响面分析
分析新功能将会：
- **需要新增的文件/模块**：列出新文件清单
- **需要修改的现有文件**：列出每个文件需要改什么（越少越好）
- **完全不需要改动的文件**：确认它们不受影响
- **可能的风险点**：哪些改动有破坏现有功能的风险

### 2.3 输出：影响评估报告

```markdown
## 影响评估报告

### 新增文件
- `path/to/new-file.ts` — 职责说明

### 需修改的现有文件
- `path/to/existing.ts` — 修改内容概要，风险等级（低/中/高）

### 不受影响的模块
- [列出核心模块，确认不受影响]

### 风险点
- [风险描述] — [缓解措施]
```

等待 Boss 确认后进入设计阶段。

## 第三阶段：扩展设计

### 3.1 设计原则

**高内聚**：
- 新功能代码集中在独立的模块/目录中
- 新模块内部的代码紧密相关，共同完成一个清晰的职责
- 避免将新逻辑分散到多个不相关的现有文件中

**低耦合**：
- 新模块通过**明确定义的接口**与现有代码交互
- 优先使用依赖注入、事件机制、回调等松耦合方式
- 避免直接修改现有类/函数的内部逻辑
- 如果必须修改现有文件，使用**扩展点模式**（钩子、插件、中间件、装饰器等）

**不影响已有功能**：
- 新增代码是**加法操作**，尽量不改已有代码
- 如果必须改现有代码，改动必须**最小化**且**向后兼容**
- 每一步改动后运行原有测试，确保不破坏

### 3.2 常用扩展模式

根据项目架构，选择合适的扩展模式：

| 场景 | 推荐模式 | 说明 |
|---|---|---|
| 添加新 API 端点 | 新增路由文件 + 新增处理函数 | 不修改已有路由 |
| 添加新页面/组件 | 新增组件文件 + 在路由配置中注册 | 已有组件零修改 |
| 添加新业务模块 | 新增独立目录 + 通过接口注册 | 模块完全自治 |
| 扩展已有实体 | 装饰器/Mixin/继承 | 不修改原始类 |
| 添加横切关注点 | 中间件/拦截器/AOP | 不侵入业务代码 |
| 添加新数据模型 | 新增 Model 文件 + 迁移脚本 | 不改已有 Model |

### 3.3 设计方案展示
分段向 Boss 展示扩展设计：
- 新增模块的结构和职责
- 与现有代码的交互方式
- 接口定义
- 数据流向

每段之后询问 Boss 是否认可。

## 第四阶段：编写扩展计划

遵循 `writing-plans.md` 规则编写计划，但额外要求：

- 每个任务**必须标注**是"新增"还是"修改"操作
- "修改"类任务必须标注风险等级
- 每个任务完成后必须运行**原有测试套件**确认无回归
- 任务顺序：先新增独立模块 → 再写新模块的测试 → 最后接入现有系统

---

## 第五阶段：执行扩展

遵循 `executing-plans.md` 规则执行，但额外要求：

### 安全执行原则
1. **先新增，后连接** — 先创建完全独立的新模块并测试通过，最后再与现有系统集成
2. **每步回归** — 每个任务完成后运行原有测试套件
3. **最小接触** — 对现有文件的每一处修改都要明确理由
4. **可回滚** — 确保每一步都可以安全回滚

### 回归测试检查点
```bash
# 每个任务完成后执行
# Git 项目
git stash list  # 记录当前状态

# SVN 项目
svn status      # 检查变更文件

# 运行原有测试
npm test        # 或对应的测试命令
```

如果原有测试出现失败：
1. **立即停止**
2. 分析失败原因
3. 向 Boss 报告
4. 等待 Boss 指示后继续
