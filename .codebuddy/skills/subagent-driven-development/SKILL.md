# 子代理驱动开发

Superpowers 的核心创新。每个任务派遣一个**全新子代理**（干净上下文），完成后经过**两阶段对抗性审查**。这是对"上下文污染"问题的根本性解决——传统的 AI 编程对话随着上下文积累，代理的注意力和判断力会逐渐退化。

## ⚠️ 铁律提醒

- 每次回复先称呼 **Boss**
- 不确定的设计决策**必须先问 Boss**
- **不写兼容性代码**，除非 Boss 主动要求

## 核心设计理念

**制度化的不信任。** 实现者和审查者是不同角色，审查者被明确告知对实现者的报告持怀疑态度。这种机制使 AI 能够**连续自主工作数小时而不偏离计划**。

## 完整工作流

```
加载计划
  ↓
为当前任务创建全新子代理（干净上下文）
  ↓
子代理独立实现任务（遵循 TDD）
  ↓
第一阶段审查：spec-reviewer（规格符合性）
  ├── 通过 → 进入第二阶段
  └── 不通过 → 退回子代理修复 → 重新审查
       ↓
第二阶段审查：code-reviewer（代码质量）
  ├── 通过 → 标记任务完成
  └── 不通过 → 退回子代理修复 → 重新审查
       ↓
继续下一任务
  ↓
所有任务完成后：全局代码审查
```

---

## 1. 任务分发：子代理 Prompt 模板

为每个任务派遣全新子代理时，提供以下完整上下文：

```markdown
## 任务规格

**任务编号**: [N]
**任务标题**: [来自计划]
**文件路径**: [需要创建或修改的文件]

### 详细要求
[从计划中提取的完整任务描述]

### 验证标准
[如何验证任务完成]

### 约束
- 遵循三条铁律
- 严格 TDD：先写失败测试 → 最小实现 → 重构
- 只修改规格范围内的文件
- 不引入规格外的功能（YAGNI）
- 不编写兼容性代码
- 完成后更新三层代码自文档（头部注释 + CONTEXT.md）

### 项目上下文
[相关模块的 CONTEXT.md 内容]
[相关文件的头部三行注释]
```

**关键：每个子代理都是全新上下文。** 不要在同一个会话中连续执行多个任务——上下文积累会导致注意力退化。

---

## 2. 两阶段对抗性审查

### 第一阶段：规格符合性审查（spec-reviewer）

调用 `spec-reviewer` 子代理，提供：
- 原始任务规格
- 实现者的完成报告
- 实现的代码 diff

**spec-reviewer 被明确告知：不要信任实现者的报告。他们完成得可疑地快。独立验证每一项规格要求。**

审查要点：
- 实现是否**精确匹配**规格？不多不少
- 是否有遗漏的功能点？
- 是否有超出规格的实现？（YAGNI 违反）
- 测试是否真的覆盖了规格要求？（自己运行测试验证）
- 是否意外引入了兼容性代码？（铁律三违反）

**审查不通过 → 退回实现者，附带具体修复清单 → 修复后重新完整审查**

### 第二阶段：代码质量审查（code-reviewer）

调用 `code-reviewer` 子代理，提供：
- 实现的代码 diff
- 第一阶段审查通过的确认

审查要点：
- 代码简洁易读？命名有意义？
- 有重复代码？错误处理完善？
- 有安全漏洞？输入验证到位？
- 测试质量如何？
- 高内聚低耦合？
- 三层文档是否已更新？

反馈按严重性分级：
- **🔴 严重**（必须修复）— 功能错误、安全漏洞、数据丢失
- **🟡 警告**（应该修复）— 代码异味、可维护性、性能
- **🔵 建议**（考虑改进）— 更好的命名、模式、文档

**🔴 严重问题阻止继续。退回修复后重新审查。**

---

## 3. 问题处理

- 🔴 严重问题：**阻塞**。退回子代理修复，修复后重新完整审查
- 🟡 警告：要求修复，修复后确认
- 🔵 建议：记录到改进列表，不阻塞
- 不确定如何处理的问题：**询问 Boss**
- 审查分歧（实现者不同意审查意见）：**升级给 Boss 决策**

## 4. 进度跟踪

每个任务完成后：
1. 向 Boss 更新进度
2. **同步更新 `docs/progress.md`**（记录完成状态、修改的文件、遇到的错误）
3. 如有研究发现或技术决策，更新 `docs/findings.md`

```
Boss，任务进展更新：

✅ 任务 1: [描述] — 已完成（规格审查✓ 质量审查✓）
✅ 任务 2: [描述] — 已完成（规格审查✓ 质量审查✓）
🔄 任务 3: [描述] — 规格审查未通过，修复中（原因：缺少边界校验）
⏳ 任务 4-8: 等待执行

总进度：2/8 已完成
持久化文件：docs/progress.md 已更新
```

## 5. 全局代码审查

**所有任务完成后**，进行一次全局审查：

1. 审查所有任务的代码作为一个整体
2. 检查模块之间的集成是否正确
3. 运行完整测试套件
4. 检查三层文档是否全部更新
5. 向 Boss 提交最终审查报告

---

## 何时使用

- 实施计划包含 **3 个以上**任务
- 任务之间依赖关系清晰
- 需要长时间自主工作
- Boss 不需要在每个任务间介入（与 `executing-plans` 的批次模式相比，子代理模式更自动化）

## 何时使用 executing-plans 替代

- Boss 需要在每批任务后手动审查
- 任务之间依赖复杂，需要人工判断执行顺序
- 项目风险较高，需要更多人工介入
