# 根因追溯技术（Root Cause Tracing）

本文档是 `systematic-debugger` 子代理的辅助材料，提供系统化的向后追溯方法论。

---

## 核心思想：从症状向源头反向追溯

大多数 bug 的表面症状和根因之间隔着多层间接调用。直接在症状位置修复往往只是掩盖问题（"消炎药"），真正的修复需要找到最初的病因（"根治"）。

```
根因（数据源的校验缺失）
  ↓ 层层传递
中间层 1（错误数据存入数据库）
  ↓
中间层 2（查询返回脏数据）
  ↓
中间层 3（业务逻辑用脏数据计算）
  ↓
症状（页面显示错误金额）  ← 你最先看到的是这里
```

**从症状向上追溯，直到找到"在此之前一切正常"的分界点——那就是根因位置。**

---

## 五步追溯法

### Step 1：锚定症状

明确描述当前的错误症状。越精确越好：

```
❌ "页面出错了"
✅ "订单详情页的应付金额显示为 -15.00，期望显示 85.00"
```

### Step 2：找到症状的直接上游

问："这个错误值是从哪来的？"

```bash
# 在代码中找到显示金额的位置
grep -rn "finalAmount\|应付金额" src/views/order/

# 找到赋值位置
# → OrderDetail.vue line 42: {{ order.finalAmount }}
# → 来自 API 返回: GET /api/orders/:id
```

### Step 3：逐层回溯数据流

对每一层问同样的问题："这个值是从哪来的？怎么计算的？"

```
前端显示 order.finalAmount = -15.00
  ↑ API返回
后端 OrderController.getOrder() → 返回 order.finalAmount
  ↑ Service计算
OrderService.calculateFinal() → amount - discount = 100 - 115 = -15
  ↑ discount从哪来？
DiscountService.getDiscount(userId) → 返回 115（应该是 15）
  ↑ 为什么返回 115？
数据库 t_discount 表 → discount_value = 115
  ↑ 谁写入的？
PromotionJob.createDiscount() → 写入时 discount_value = price * rate = 100 * 1.15
  ↑ ← 根因：rate 字段含义理解错误（以为是折扣率15%，实际存的是1.15倍率）
```

### Step 4：验证根因

对找到的根因做两个验证：

1. **正向验证**：从根因正向推导，能否完整解释所有症状？
2. **排除验证**：修复根因后，症状是否消失？其他正常功能是否不受影响？

### Step 5：确定修复位置

根因位置 ≠ 修复位置。有时根因在 A 处，但最佳修复在 B 处：

```
根因：PromotionJob 写入时理解错误
修复选项：
A) 修复 PromotionJob 的写入逻辑 + 修复历史脏数据
B) 修复 DiscountService 的读取逻辑（除以100转换）
C) 同时修复 A 和 B（防御性编程）

→ 推荐 A（修复源头） + 临时 B（处理已有脏数据） + 数据修复脚本
```

## 常用追溯工具

| 追溯方向 | 工具 |
|---|---|
| 数据流回溯 | 断点调试、日志追踪、grep 搜索赋值链 |
| 时间线回溯 | `git log`/`svn log`、`git bisect`、部署记录 |
| 调用链回溯 | 堆栈追踪、调用图（IDE 工具）、`grep -rn "函数名"` |
| 状态变化回溯 | 数据库审计日志、Redux DevTools、状态快照对比 |

## 常见追溯陷阱

1. **在中间层停下来就修**：找到一个"看起来有问题"的中间层就动手修，实际根因还在更上游
2. **假设"之前能用所以没问题"**：最近没改过的代码也可能是根因——环境变化、数据变化都可能触发潜在 bug
3. **只追溯一条路径**：并发场景下可能有多条数据流交叉，需要同时追溯多条路径
4. **追溯太远**：追溯到宇宙大爆炸也没用。当你找到"在此之前一切正常"的分界点，就是根因
