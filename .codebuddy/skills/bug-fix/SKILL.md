# 问题单修改方法论

基于问题单信息，通过上下文完整读取、精准定位、全面修改、严格验证的全流程方法论。

## 核心原则

1. **上下文完整**：修改前必须充分读取上下文，包括主文件完整内容、相关文件、数据定义、方法定义、相似代码等
2. **全面修改**：识别所有需要修改的位置，不遗漏引用点、相似点、对称点
3. **精准修改**：只解决当前问题，不修改无关代码
4. **最小化变更**：确保完整性的前提下，修改范围最小化
5. **必须确认**：不确定的方案必须用户确认
6. **遵循用户方案**：用户提供方案时，基于充分上下文验证正确性和完整性，补充遗漏
7. **修改前验证**：距离上次读取超过5条消息，必须重新读取文件确认内容
8. **修改后验证**：验证语法、逻辑、一致性和完整性，检查遗漏

## 上下文分层读取策略

### 必读层级（必须完成）

- **主文件**：完整读取（>2000行则至少读取问题前后各200行+文件头部声明）
- **方法上下文**：完整读取涉及的方法/函数定义（参数、返回值、内部逻辑）
- **数据定义**：相关变量、数据结构、常量、配置项定义
- **导入依赖**：文件头部的导入/引入语句

### 扩展层级（根据问题类型决定）

- **调用关系**：调用方和被调用方的相关代码
- **相关工具**：工具函数、常量定义、配置文件、辅助模块
- **接口定义**：API接口定义、类型定义文件、接口契约
- **相似逻辑**：搜索相似的代码模式（相同方法调用、相同逻辑结构）

### 上下文完整性检查清单

在生成修改方案前，必须确认：

- ✅ 主文件完整上下文已读取
- ✅ 涉及的方法/函数/类/模块定义已读取
- ✅ 相关数据结构、变量、常量、配置项已读取
- ✅ 相关导入依赖已识别
- ✅ 相关引用文件已读取（调用方、被调用方、依赖模块）
- ✅ 相似代码模式已搜索和对比
- ✅ 所有可能需要修改的位置已标记

## 全面修改点识别

### 必查检查项

1. **主修改点**：问题最直接相关的代码位置
2. **相关引用点**：所有引用了相关变量/方法/类/模块的位置
3. **相似代码点**：项目中相似代码模式，可能需统一修改
4. **对称逻辑点**：存在对称或相反逻辑的位置
5. **关联数据点**：相关数据定义、常量、配置项、枚举值
6. **类型定义点**：相关类型定义（TypeScript、Java、Go、Python等）
7. **注释文档点**：相关注释、文档注释（JSDoc、JavaDoc等）

### 排查策略

- 从问题点向外扩散，逐层检查
- 调用链：谁调用了问题方法 → 问题方法调用了谁
- 数据链：数据来源 → 数据处理 → 数据使用
- 事件链：事件触发 → 事件监听 → 事件处理
- 依赖链：模块/类的依赖关系

## 修改前验证流程

1. **重新读取文件**：距离上次读取超过5条消息，必须重新读取
2. **精确匹配old_str**：确认唯一性，不唯一则扩大上下文范围；保留完整缩进、空格、换行、标点
3. **验证修改点完整**：确认无遗漏，每个old_str和new_str准确无误
4. **每个文件连续修改不超过3次**，超过则重新读取文件

## 修改后验证流程

1. **语法验证**：调用 `read_lints` 检查linter错误，有则立即修复
2. **逻辑验证**：检查修改后代码逻辑正确性，是否遗漏修改点，是否破坏原有功能
3. **一致性验证**：变量名/方法名一致、代码风格统一、注释是否需更新
4. **遗漏检查**：重新搜索相关关键词，确认所有位置都已处理

## 常见错误（必须规避）

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 上下文读取不足，只读问题附近片段 | 误判原因，引入新问题 | 主文件完整读取+相关定义+依赖组件 |
| 修改点识别不全，只改主位置 | 修改不完整，不一致 | search_content搜索所有引用，统一修改 |
| 修改前未重新读取文件 | old_str匹配失败，覆盖最新修改 | 超过5条消息必须重新读取 |
| 过度依赖用户方案，不验证 | 方案有误或不完整 | 基于上下文验证正确性和完整性 |
| 修改后不验证 | 语法错误、逻辑错误、遗漏 | read_lints检查+逻辑验证+遗漏检查 |

## 严格约束

✅ **必须遵守**

- 上下文读取必须充分完整（主文件、相关文件、数据定义、方法定义、相似代码）
- 修改点识别必须全面（引用点、相似点、对称点）
- 修改前重新读取文件（距上次读取超过5条消息）
- 修改后验证语法、逻辑、一致性、完整性
- 不修复无关问题，发现后先询问用户
- 不猜测，信息不明确必须确认
- 保持代码风格和格式不变
- 用户提供方案时，必须验证正确性和完整性

❌ **禁止行为**

- 上下文不充分就给出修改方案
- 识别修改点不全面，遗漏相关位置
- 未经用户确认就修改
- 修改无关代码、添加新功能、重构优化（除非是问题原因）

## 特殊情况处理

| 情况 | 处理方式 |
|------|---------|
| 未提供文件 | 自动分析项目定位文件 |
| 信息不完整 | 列出缺失信息，询问补充或推测 |
| 无法定位 | 说明搜索结果，提供可能原因，建议补充信息 |
| 风险过高 | 明确风险点，建议测试环境验证 |
| 图片模糊 | 提示提供更清晰截图或网址替代 |

## 输入建议

- ✅ 问题单已有修改方案或结论：直接提供，AI将验证并执行
- ⚠️ 不确定的定位原因：建议不提供，防止干扰独立分析
- ✅ 明确代码文件：一并提供，提升定位速度
- ⚠️ 接口参数修改：必须提供接口入参、返参示例
- 🎯 提供期望结果：明确描述修改后应有的效果
