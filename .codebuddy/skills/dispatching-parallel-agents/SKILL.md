# 并行代理分发

当面对多个独立任务时，同时派遣多个子代理并行工作，大幅缩短开发时间。

## ⚠️ 铁律提醒

- 每次回复先称呼 **Boss**
- 不确定的设计决策**必须先问 Boss**
- **不写兼容性代码**，除非 Boss 主动要求

## 何时使用

当计划中存在 **2 个以上可独立执行的任务** 时，评估是否可以并行分发。

## 决策流程

```
收到多个任务
  ↓
任务之间是否有数据依赖？
  ├── 全部独立 → 全部并行分发
  ├── 部分依赖 → 独立的并行，有依赖的串行
  └── 全部依赖 → 按顺序串行执行
       ↓
并行任务是否共享文件？
  ├── 不共享 → 直接并行分发
  └── 共享同一文件 → 必须串行或拆分文件
       ↓
并行分发任务
  ↓
等待所有任务完成
  ↓
合并结果 + 解决冲突
  ↓
集成测试验证
```

## 独立性判断标准

两个任务可以并行，当且仅当：

1. **不修改同一文件** — 否则会产生合并冲突
2. **不依赖对方的输出** — 任务 B 不需要任务 A 的产物作为输入
3. **不共享可变状态** — 不操作同一数据库表的同一字段、同一缓存键等
4. **测试互不干扰** — 各自的测试可以独立运行

## 分发流程

### 1. 任务分组

将计划中的任务按依赖关系分为若干**并行组**：

```
并行组 1: [任务A, 任务B, 任务C]  ← 三个完全独立，可同时执行
   ↓ 全部完成后
并行组 2: [任务D, 任务E]          ← 依赖组1的结果，但D和E之间独立
   ↓ 全部完成后
串行: 任务F                       ← 依赖所有前序任务
```

### 2. 为每个并行任务分发子代理

每个子代理接收：
- 完整的任务规格
- 项目上下文（相关的 CONTEXT.md）
- 三条铁律
- **明确的文件边界**（只允许修改哪些文件）
- 当前并行组中其他任务的摘要（避免冲突）

### 3. 汇总与集成

所有并行任务完成后：
1. 检查是否有意外的文件冲突
2. 运行集成测试
3. 如有冲突，向 Boss 报告并请求决策
4. 更新三层代码自文档

## 风险控制

- **最大并行度**：同时分发不超过 **5 个**子代理（超过则分批）
- **超时机制**：单个子代理如果长时间无响应，标记为异常
- **冲突预防**：分发时明确各子代理的文件修改范围，禁止越界
- **回滚准备**：每个并行组开始前确保有干净的版本控制基线

## 向 Boss 报告

并行分发前：
```
Boss，计划中的 12 个任务可以分为 3 个并行组：
- 并行组 1: 任务 1/2/3（完全独立，可同时执行）
- 并行组 2: 任务 4/5（依赖组1，但互相独立）
- 串行: 任务 6-12（有链式依赖）

预计并行执行可节省约 40% 时间。
Boss 确认这个分组吗？
```

## 不适用场景

- 所有任务都操作同一个核心文件
- 任务之间有复杂的双向依赖
- Boss 要求严格按顺序执行和审查
