# C/C++ 安全编码规范

> **版本**：V2.0 &nbsp;|&nbsp; **适用范围**：所有基于 C/C++ 语言开发的产品  
> **级别定义**：【原则】必须遵守的指导思想 ·【规则】必须遵守的强制约定 ·【建议】应加以考虑的推荐约定

---

## 1 通用安全原则

### 1.1 【原则】对外部输入进行严格校验

外部输入（用户输入、外部接口、配置文件、网络数据、环境变量等）在以下场景必须校验其合法性：

- 作为循环条件、数组索引、指针偏移
- 作为内存分配尺寸参数
- 作为格式化字符串
- 参与业务逻辑（命令执行参数、SQL拼装等）

**校验策略（优先级从高到低）**：

| 策略 | 说明 |
|------|------|
| 白名单校验 | 仅接受已知合法字符集，**优先推荐** |
| 数据长度校验 | 校验输入数据长度是否在合理范围内 |
| 数据范围校验 | 校验数值是否在有效值域内 |
| 归一化处理 | 对输入做编码规范化，防止字符转义绕过 |
| 命令字符检测 | 检测 `|`、`||`、`&`、`&&`、`;`、`$` 等命令注入字符 |

### 1.2 【原则】禁止在日志中保存敏感信息

日志中不得包含明文或密文口令、密钥、工号等敏感信息。若因特殊原因必须记录，须以 `***` 脱敏处理。

### 1.3 【原则】及时清除可复用资源中的敏感信息

存储口令、密钥的变量使用完毕后，必须用 `memset_s` 或等效安全函数进行显式覆盖清零。涉及的可复用资源包括堆、栈、数据段及数据库映射缓存。

> **注意**：不要使用普通 `memset`，编译器可能因优化而将其删除。应使用 `memset_s` 或 `volatile` 指针清零方案。

### 1.4 【原则】使用经过验证的标准加密算法

禁止使用私有算法或弱加密算法（DES、SHA1、MD5、RC4 等）。推荐算法与最低密钥长度：

| 算法类型 | 推荐算法 | 最低密钥长度 |
|----------|----------|-------------|
| 对称加密 | AES | 192 位 |
| 非对称加密 | RSA | 2048 位 |
| 数字签名 | DSA | 1024 位 |
| 哈希算法 | SHA-256 | 256 位 |

基于哈希算法的口令存储必须加入盐值（Salt）。

### 1.5 【原则】遵循最小权限原则

程序运行时只分配完成当前任务所需的最小权限。权限撤销须遵循正确的顺序（先撤销用户层面权限，再撤销组层面权限），并在撤销后校验是否成功。

### 1.6 【原则】禁止硬编码口令、密钥等敏感信息

口令、密钥等敏感数据不得直接写入源代码，应通过安全的配置管理或密钥管理系统获取。

### 1.7 【原则】删除无效代码与未使用变量

不产生任何效果的代码和未使用的变量可能隐藏逻辑错误，应当识别并清理。

## 2 字符串操作安全

### 2.1 【规则】确保有足够空间存储字符数据和 `'\0'` 结束符

分配内存或执行字符串复制时，必须为 `'\0'` 结束符预留空间。

```c
// ✗ 错误：源字符串长度可能超过目标空间
char dst[128];
strcpy(dst, argv[1]);

// ✓ 正确：根据源字符串长度分配并手动添加终止符
if (strlen(argv[1]) < 40) {
    char *dst = (char *)malloc(strlen(argv[1]) + 1);
    if (dst != NULL) {
        strncpy(dst, argv[1], strlen(argv[1]));
        dst[strlen(argv[1])] = '\0';
    }
}
```

### 2.2 【规则】字符串操作过程中确保有 `'\0'` 结束符

`strncpy` 等带 `n` 版本的函数在源字符串超长时会截断 `'\0'`，必须手动补齐。`strlen` 不计入 `'\0'`，配合 `memcpy` 使用时同样需要注意。

```c
char dst[5];
char src[] = "0123456789";
strncpy(dst, src, sizeof(dst));
dst[sizeof(dst) - 1] = '\0';  // 必须手动补齐
```

### 2.3 【规则】复制数据到固定长度内存前必须检查边界

对来自 `gets()`、`getenv()`、`scanf()` 等不可预测长度的数据，复制前须先获取并检查其长度。

### 2.4 【规则】避免源指针与目标指针指向内存重叠区

`memcpy`、`strcpy`、`sprintf` 等函数的源与目标重叠会导致未定义行为。若确需操作重叠内存，应使用 `memmove`。

### 2.5 【规则】禁止使用无长度限制的字符拷贝函数

禁止直接使用 `strcpy`、`strcat`、`sprintf`、`gets` 等函数。替代方案：

| 禁用函数 | 替代方案 |
|----------|----------|
| `strcpy` | `strncpy` + 手动补 `'\0'` 或 `_s` 安全版本 |
| `strcat` | `strncat` + 长度检查 |
| `sprintf` | `snprintf` |
| `gets` | `fgets` |

> **注意**：MSVC 的 `_snprintf` 在缓冲区满时不保证 `'\0'` 结尾。建议预留 1 字节并手动置零。

```c
char buf[101] = {0};
snprintf(buf, sizeof(buf) - 1, "foobar %s", str);
```

### 2.6 【规则】字符串数组传入函数时须同时传入长度

字符串以指针传入函数时，必须将其长度作为参数一并传入。若字符串内容不会改变，指针须加 `const` 修饰。

## 3 格式化输出安全

### 3.1 【规则】格式化参数与实参类型必须匹配

格式字符与参数的类型不匹配会导致未定义行为，可能引发格式化漏洞或程序崩溃。禁止将结构体整体作为格式化参数。

### 3.2 【规则】格式化参数与实参个数必须匹配

拼装调试信息时尤其注意，复制粘贴易导致格式化参数数量不一致。

### 3.3 【规则】禁止以用户输入构造格式化字符串

```c
// ✗ 危险：用户可通过 %s%s%s 读取栈内存
printf(input);

// ✓ 安全：使用显式格式化参数
printf("%s", input);
```

### 3.4 【建议】使用精度说明符限制字符串拷贝长度

```c
sprintf(buffer, "Usage: %.100s argument\n", argv[0]);
```

### 3.5 【建议】防止泄露指针值

`%p` 输出的地址信息仅供程序内部使用，不得输出到外部，以免破坏 ASLR 防护。

### 3.6 【建议】Debug 信息不应包含内存地址、内存内容等敏感值

推荐输出有特定含义的错误码，替代实际数据值。生产环境应关闭 Debug 输出。

## 4 整数安全

### 4.1 【规则】确保无符号整数运算不会反转

无符号整数加法或乘法的结果若超出表示范围，会发生回绕（Wrap-around），须在运算前检查。

### 4.2 【规则】确保有符号整数运算不会溢出

有符号整数溢出属于未定义行为，可能被编译器优化产生预期外结果，须提前做范围检查。

### 4.3 【规则】确保整型转换不会出现截断或符号错误

将大类型赋值给小类型、有符号与无符号之间的隐式转换均可能产生截断或符号反转。在比较或赋值为更大类型前，须先用目标类型进行求值。

### 4.4 【规则】检查除以零异常

对任何可能为零的除数变量，运算前必须进行非零检查。

### 4.5 【规则】防止 Off-By-One 错误

循环边界、数组索引等场景应仔细检查是否存在差一错误。

### 4.6 【规则】注意大小端问题

跨平台或网络传输场景中须确认并正确处理字节序。

### 4.7 【建议】避免对有符号整数进行位操作

位操作在有符号整数上的行为依赖于编译器实现，可能产生非预期结果。

### 4.8 【建议】比较数据大小时加上最小/最大值校验

防止攻击者构造边界值绕过检查逻辑。

## 5 内存管理安全

### 5.1 【规则】禁止引用未初始化的内存

所有变量和动态分配的内存在使用前必须进行初始化。

### 5.2 【规则】禁止访问已释放的内存（Use-After-Free）

指针释放后必须立即置为 `NULL`，禁止继续使用。

### 5.3 【规则】禁止重复释放内存（Double Free）

对同一地址重复调用 `free()` 会导致堆损坏，须通过置空指针或严谨的生命周期管理来避免。

### 5.4 【规则】校验内存申请大小的合法性

内存分配大小若来源于外部输入，必须对其进行范围校验，防止整数溢出导致分配异常小的缓冲区。

### 5.5 【规则】禁止释放非动态申请的内存

不得对栈变量、全局变量或非 `malloc`/`new` 分配的内存调用 `free`/`delete`。

### 5.6 【规则】禁止返回栈上变量的地址

函数返回后栈帧失效，对应地址不可再访问。

### 5.7 【规则】防止越界写（Buffer Overflow）

向缓冲区写入数据前必须检查目标空间是否足够，包括前向越界与后向越界。

### 5.8 【规则】检查空指针

对指针进行解引用前，必须判断其是否为 `NULL`。

### 5.9 【规则】释放后置空指针

```c
free(ptr);
ptr = NULL;
```

### 5.10 【规则】防止错误的类型转换（Type Confusion）

对 `void*`、联合体等进行类型转换时，须确保目标类型与实际类型一致。

### 5.11 【规则】智能指针使用安全（C++）

- 使用 `std::make_shared` / `std::make_unique` 替代裸 `new`
- 禁止对同一裸指针构造多个 `shared_ptr`
- 禁止在容器中存放 `auto_ptr`

### 5.12 【建议】避免使用 `alloca` 申请内存

`alloca` 在栈上分配，大小不可控时极易导致栈溢出。

### 5.13 【建议】校验内存相关函数的返回值

`malloc`、`realloc`、`calloc` 等函数失败时返回 `NULL`，必须检查返回值后再使用。

## 6 禁用不安全函数

### 6.1 【规则】禁止使用未显式指明目标缓冲区大小的字符串操作函数

| 禁用函数 | 安全替代 |
|----------|----------|
| `strcpy` | `strncpy` / `strcpy_s` |
| `strcat` | `strncat` / `strcat_s` |
| `sprintf` | `snprintf` / `sprintf_s` |
| `vsprintf` | `vsnprintf` / `vsprintf_s` |
| `gets` | `fgets` |
| `scanf("%s")` | `scanf("%Ns")` 限定长度 |

### 6.2 【规则】禁止调用 OS 命令解析器执行命令

禁止使用 `system()`、`popen()` 等函数执行外部命令，以防止命令注入。确有需要时，须对输入进行严格白名单校验和危险字符过滤。

### 6.3 【规则】禁止使用 `std::ostrstream`

使用 `std::ostringstream` 替代，前者因不自动管理内存已被弃用。

### 6.4 【规则】C++ 中须使用标准库替代 C 字符串函数

优先使用 `std::string`、`std::vector` 等高级封装替代原始指针和动态数组操作。

## 7 文件操作安全

### 7.1 【规则】使用 `int` 类型接收字符输入/输出函数的返回值

`fgetc()`、`getc()` 等函数返回值为 `int`（含 `EOF` 标志），使用 `char` 接收会导致判断 `EOF` 失败。

### 7.2 【规则】创建文件时必须显式指定访问权限

禁止使用默认权限创建文件，须通过 `umask` 或显式设置 `mode` 参数限定为最小必要权限。

### 7.3 【规则】文件路径验证前须进行标准化

使用 `realpath()` 等函数将路径标准化后再进行校验，防止 `../` 等目录遍历攻击。

### 7.4 【建议】使用文件描述符代替文件名，避免 TOCTOU 竞态条件

检查文件属性（如权限、存在性）后，应使用同一文件描述符进行后续操作，防止检查与使用之间文件被替换。

## 8 C++ 特有安全规则

### 8.1 【规则】禁止切分多态类对象

将派生类对象赋值给基类值类型变量会发生对象切割（Object Slicing），丢失派生类信息。应通过指针或引用传递多态对象。

### 8.2 【规则】可能被继承的类的析构函数必须为 `virtual`

否则通过基类指针 `delete` 派生类对象时不会调用派生类析构函数，造成资源泄漏。

### 8.3 【规则】数组 `delete` 须使用 `delete[]`

```cpp
int *arr = new int[10];
delete[] arr;  // ✓
// delete arr;  // ✗ 未定义行为
```

### 8.4 【规则】避免 `delete this`

`delete this` 后对象已失效，但调用方可能仍持有指针，极易引发 Use-After-Free。

### 8.5 【规则】禁止在公共接口中返回私有数据地址

返回内部数据的指针或引用会破坏封装性，外部可直接修改内部状态。应返回副本。

### 8.6 【规则】STL 迭代器使用安全

- 迭代器使用前须确保有效
- 容器 `erase()` 后须更新迭代器
- 不同容器的迭代器不可混用

## 9 其他安全规则

### 9.1 【规则】禁止使用 `rand()` 产生安全用途的伪随机数

`rand()` 的随机性不足以满足安全需求。应使用操作系统提供的加密安全随机数生成器（如 Linux 的 `/dev/urandom`、Windows 的 `CryptGenRandom`）或 C++ 的 `<random>` 库中的安全引擎。

### 9.2 【规则】禁止存储 `getenv()` 返回的字符串指针

`getenv()` 返回的指针指向的内容可能被后续的 `putenv()` / `setenv()` 修改。如需保留环境变量值，须立即进行深拷贝。

### 9.3 【规则】多线程环境禁止使用线程不安全函数

如 `strtok`、`localtime`、`asctime` 等非线程安全函数，应使用其 `_r`（POSIX）或 `_s`（MSVC）版本替代。

### 9.4 【规则】`switch` 语句须包含 `default` 分支

即使所有枚举值均已处理，仍应提供 `default` 分支以应对未来变更。

### 9.5 【规则】确保变量的线程安全性

多线程共享变量须使用互斥锁、原子操作或其他同步机制保护。

### 9.6 【建议】编译时使用最高警告等级

建议开启 `-Wall -Wextra -Werror`（GCC/Clang）或 `/W4 /WX`（MSVC），将警告视为错误。

### 9.7 【建议】防止敏感数据清零代码被编译器优化删除

使用 `memset_s`、`SecureZeroMemory`（Windows）或 `volatile` 指针方案：

```c
void secure_zero(void *ptr, size_t len) {
    volatile unsigned char *p = (volatile unsigned char *)ptr;
    while (len--) *p++ = 0;
}
```

### 9.8 【建议】注意 TOCTOU 竞态条件与 Signal Handler 竞态条件

检查与使用之间存在时间窗口时，攻击者可能利用该窗口篡改状态。应尽量使用原子操作或合适的同步机制。
